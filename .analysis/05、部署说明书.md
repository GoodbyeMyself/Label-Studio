# Label Studio 部署说明书

## 1. 文档目标

这份文档面向部署和运维场景，重点说明：

- 当前仓库支持哪些部署方式
- 每种部署方式适合什么场景
- 当前默认容器结构是什么
- 生产部署前需要确认哪些关键项
- 常见部署问题如何规避

本文优先基于当前仓库中的：

- `Dockerfile`
- `docker-compose.yml`
- `docker-compose.mysql.yml`
- `docker-compose.minio.yml`
- `deploy/`

## 2. 当前仓库支持的部署方式

从仓库配置和文档看，当前工程支持以下几类部署：

1. 源码本地运行
2. Docker 单容器运行
3. Docker Compose 默认部署
4. Docker Compose + MySQL
5. Docker Compose + MinIO
6. 自定义镜像部署
7. 云上部署参考
   - Heroku
   - Azure
   - Cloud Run
   - Kubernetes / Helm

如果你们要做正式环境部署，建议优先从第 3 类开始，也就是 Docker Compose 默认部署。

## 3. 默认容器架构

## 3.1 `docker-compose.yml` 默认包含的服务

默认 Compose 包含三个核心服务：

- `nginx`
- `app`
- `db`

它们的职责如下：

### `app`

职责：

- 运行 Label Studio 应用
- 通过 `label-studio-uwsgi` 启动
- 处理业务请求
- 连接数据库

关键点：

- 内部暴露 `8000`
- 使用 `uWSGI`
- 容器启动时会执行 entrypoint 脚本

### `nginx`

职责：

- 对外暴露 HTTP/HTTPS 入口
- 代理转发到 `app`
- 提供静态资源访问
- 支持文件下载代理

关键端口映射：

- 主机 `8080 -> 容器 8085`
- 主机 `8081 -> 容器 8086`

一般对外访问：

- `http://localhost:8080`

### `db`

职责：

- 提供 PostgreSQL 数据库

当前默认镜像：

- `pgautoupgrade/pgautoupgrade:17-alpine`

## 4. 推荐部署路径

## 4.1 路径一：源码本地运行

适合：

- 个人调试
- 本地开发
- 快速排查

不适合：

- 正式生产环境
- 团队共享测试环境

## 4.2 路径二：Docker 单容器

适合：

- 演示
- 快速试用
- 轻量单机

启动：

```bash
docker pull heartexlabs/label-studio:latest
docker run -it -p 8080:8080 -v $(pwd)/mydata:/label-studio/data heartexlabs/label-studio:latest
```

特点：

- 部署快
- 默认简单
- 更适合 PoC，不是推荐的正式环境方案

## 4.3 路径三：Docker Compose 默认部署

适合：

- 测试环境
- 中小规模正式环境
- 私有化部署

启动：

```bash
docker compose up -d
```

访问：

- `http://localhost:8080`

推荐程度：

- 当前仓库最值得优先采用的默认部署方式

## 4.4 路径四：Docker Compose + MySQL

适合：

- 团队已有 MySQL 基础设施
- 希望保持和现有 MySQL 生态一致

启动：

```bash
docker compose -f docker-compose.yml -f docker-compose.mysql.yml up -d
```

说明：

- `docker-compose.mysql.yml` 会覆盖数据库环境变量
- 默认数据库容器切换到 `mysql:8`

## 4.5 路径五：Docker Compose + MinIO

适合：

- 本地模拟 S3
- 测试对象存储接入
- 调试导入导出和存储代理

启动：

```bash
docker compose -f docker-compose.yml -f docker-compose.minio.yml up -d
```

补充说明：

- MinIO 对外端口通常是 `9000`
- Console 端口通常是 `9009`
- 仓库里也提供了与 MinIO 相关的 Prometheus 配置

## 5. Dockerfile 构建逻辑

当前仓库的 `Dockerfile` 是一个多阶段构建，核心阶段如下：

1. `frontend-builder`
   - 构建前端产物
2. `frontend-version-generator`
   - 生成前端版本信息
3. `venv-builder`
   - 安装 Python 依赖，构建虚拟环境
4. `py-version-generator`
   - 生成 Python 版本文件
5. `production`
   - 组装最终生产镜像

这意味着：

- 前端和后端产物统一打进一个最终镜像
- 最终镜像里已经带 Nginx、Python 环境、静态资源和启动脚本

本地构建命令：

```bash
docker build -t heartexlabs/label-studio:latest .
```

## 6. 容器启动过程

容器启动核心入口：

- `deploy/docker-entrypoint.sh`

根据启动参数会分不同模式：

- `nginx`
- `label-studio-uwsgi`
- `label-studio-migrate`
- 其他自定义命令

其中，`label-studio-uwsgi` 是默认应用启动模式，会：

1. 执行 `deploy/docker-entrypoint.d/app/` 下的初始化脚本
2. 等待数据库
3. 执行数据库迁移
4. 执行初始化逻辑
5. 最终启动 uWSGI

这说明：

- 默认容器并不是“只启动一个进程”那么简单
- 部署时不要随意跳过 entrypoint

## 7. 生产部署关键配置

## 7.1 数据库

生产环境建议优先：

- PostgreSQL

原因：

- 默认 Compose 已按 PostgreSQL 组织
- 项目本身对 PostgreSQL 支持成熟
- SQLite 不适合正式环境

## 7.2 数据目录

默认卷挂载常见有：

- `./mydata:/label-studio/data`
- `./postgres-data:/var/lib/postgresql/data`

建议：

- 明确业务数据、数据库数据、日志目录的持久化位置
- 不要把关键数据只放在容器临时层

## 7.3 非 root 用户权限

这是部署时最容易踩的坑之一。

仓库文档明确说明：

- 容器默认以非 root 用户运行
- 默认 UID 为 `1001`

这意味着：

- 所有需要写入的挂载目录必须有正确权限
- 否则容易出现：
  - `Permission denied`
  - 无法写入 `/label-studio/data/media`

部署前建议明确检查：

- 主机目录是否可写
- 容器用户是否有组权限

## 7.4 反向代理和静态资源

默认 Nginx 配置在：

- `deploy/default.conf`

主要职责包括：

- 反向代理到 `http://localhost:8000`
- 直接提供 `/static/`
- 提供 `/react-app/`
- 支持下载代理

如果你们已有外部网关或统一 Ingress，可以按现有 Nginx 配置进行裁剪或替换，但建议先跑通默认版本再改。

## 7.5 环境变量

生产环境不要硬编码配置，建议全部通过环境变量注入，至少包括：

- 数据库连接
- 域名/主机地址
- Session/Cookie 安全项
- 对象存储凭证
- 第三方日志、监控、Sentry 配置

## 8. 推荐部署步骤

## 8.1 测试环境部署步骤

推荐步骤：

1. 构建或拉取镜像
2. 准备数据目录
3. 检查目录权限
4. 执行 `docker compose up -d`
5. 检查容器日志
6. 打开浏览器验证页面
7. 创建测试用户和测试项目
8. 导入一批测试数据

## 8.2 生产环境部署步骤

推荐顺序：

1. 选定数据库方案
2. 规划持久化卷
3. 配置环境变量
4. 配置域名、TLS、反向代理
5. 完成镜像构建或制品发布
6. 在预发环境跑一遍
7. 验证导入、标注、导出、文件访问
8. 再切到正式环境

## 9. 部署模式建议

## 9.1 单机小规模

建议：

- Docker Compose
- PostgreSQL
- 本地卷或 NAS

适合：

- 小团队内部使用
- 数据规模不大
- 先快速上线

## 9.2 企业内网部署

建议：

- Docker Compose 或 Kubernetes
- PostgreSQL 独立部署
- 对象存储独立部署
- 统一网关或 Ingress

适合：

- 多用户协作
- 有正式数据管理要求
- 有日志审计和安全要求

## 9.3 云上部署

如果要上云：

- 可以参考仓库文档里的 Azure / Cloud Run / Helm 路径
- 但建议先在本地或内网测试环境把默认镜像跑稳

原因：

- 云部署的问题通常并不在“能否启动”
- 而在网络、卷、权限、对象存储、域名和安全策略

## 10. 常见部署问题

## 10.1 容器启动后访问不到页面

优先检查：

- `docker compose ps`
- `nginx` 和 `app` 是否都启动成功
- 主机 `8080` 端口是否被占用
- Nginx 是否已代理到 `8000`

## 10.2 数据库连接失败

优先检查：

- `db` 容器是否正常启动
- `POSTGRE_HOST` / `POSTGRE_PORT` 是否正确
- `DJANGO_DB` 是否设置为正确类型

## 10.3 挂载目录权限错误

典型报错：

- `PermissionError`
- 媒体目录无法写入

优先检查：

- 主机目录是否存在
- 挂载目录是否授予容器用户写权限
- 是否理解当前镜像是非 root 运行

## 10.4 前端资源加载异常

优先检查：

- 前端构建产物是否已进入镜像
- `/react-app/` 是否被正确代理
- 是否误改 Nginx 静态目录配置

## 10.5 上传或下载文件失败

优先检查：

- Nginx 代理配置
- `client_max_body_size`
- 文件路径和对象存储配置
- 代理下载相关 location 是否被外部网关干扰

## 11. 部署后最小验收清单

正式环境上线前，建议至少完成以下验收：

1. 能正常登录
2. 能创建项目
3. 能导入任务
4. 能打开标注页面
5. 能保存标注
6. 能导出结果
7. 如果接入对象存储，能正常读取和导出
8. 服务重启后数据不丢
9. 日志可追踪
10. 备份策略已明确

---

## 12. 推荐的生产化增强项

如果要长期稳定运行，建议逐步补齐：

1. PostgreSQL 独立高可用或托管化
2. 对象存储独立化
3. 统一日志采集
4. Sentry 或同类异常监控
5. 备份与恢复策略
6. 资源限制与容量规划
7. 预发环境回归流程

## 13. 本仓库部署建议结论

对于当前仓库，建议部署策略如下：

### 开发环境

- 本地源码运行
- 必要时配合前端 HMR

### 测试环境

- `docker compose up -d`

### 生产环境

- 优先使用 Docker Compose 或 Kubernetes
- 数据库使用 PostgreSQL
- 数据目录与对象存储做持久化
- 保留默认 entrypoint 和 Nginx 逻辑，再逐步裁剪

如果你们只是要先把平台跑起来，不建议一开始就：

- 改启动链路
- 改 Nginx 整体结构
- 改容器运行用户
- 把数据库切成 SQLite

先按默认架构跑通，是成本最低、风险最小的做法。
