# Label Studio 团队接手与二开指南

## 1. 文档目的

这份文档不是重复介绍工程结构，而是从“团队接手、持续开发、逐步上线”的角度，给出更适合实际落地的执行建议。

建议与根目录中的 `01、工程快速分析.md` 配套阅读：

- `01、工程快速分析.md`
  - 解决“这个工程是什么、有哪些模块、怎么部署”
- `02、团队接手与二开指南.md`
  - 解决“团队应该怎么接手、怎么拆任务、怎么做二开、怎么规避风险”

---

## 2. 先给结论

如果你们团队准备基于当前工程做二次开发，建议不要一上来就大改，而是按下面顺序推进：

1. 跑通本地开发环境
2. 跑通 Docker Compose 部署
3. 明确改造目标属于哪一层
4. 只选一个切入点做第一阶段改造
5. 建立最小可回归验证流程
6. 再推进页面、接口、存储、模型等后续扩展

最重要的一条是：

先把工程“跑起来、看明白、能调试”，再做架构性改造。

---

## 3. 团队接手时最容易踩的坑

## 3.1 把它当成普通后台系统

这个项目虽然有很多管理页面，但本质上不是简单后台。

它至少包含四层：

- Django 后端业务层
- React 主应用层
- 标注编辑器内核层
- 数据管理与存储接入层

如果把它误判成“只是页面多一点的后台系统”，后续就会在编辑器、任务流、存储代理、前后端联动上频繁踩坑。

## 3.2 同时改太多层

这类工程最怕：

- 前端页面在改
- 后端接口在改
- 存储逻辑也在改
- Docker 部署也顺手改
- 最后问题无法定位

建议严格控制首轮改造范围，一次只动一个主方向。

## 3.3 不区分“业务定制”和“平台能力改造”

这两类改造成本完全不同：

- 业务定制
  - 新页面
  - 新接口
  - 流程调整
  - 数据字段扩展
- 平台能力改造
  - 标注器行为调整
  - 存储代理逻辑改造
  - ML Provider 接入改造
  - 任务分发机制改造

建议团队先判断需求到底属于哪一类，再安排人力。

## 4. 推荐接手顺序

## 4.1 第一步：先跑通最小开发环境

建议先完成下面两件事：

### 本地后端跑通

```bash
pip install poetry
poetry install
make migrate-dev
make run-dev
```

### 前端开发环境跑通

```bash
cd web
yarn install --frozen-lockfile
yarn dev
```

目标不是马上开发，而是确认以下事情：

- 依赖是否完整
- 本地 Python/Node 版本是否兼容
- 前后端能否联调
- 开发命令是否可复用

## 4.2 第二步：跑通容器化部署

建议至少跑一次：

```bash
docker compose up
```

这样做的价值是：

- 看清生产默认启动方式
- 熟悉 app / nginx / db 的关系
- 提前暴露环境变量、卷挂载、端口冲突等问题

## 4.3 第三步：按职责看代码，而不是按目录乱看

推荐阅读顺序：

### 后端

1. `label_studio/core/settings/base.py`
2. `label_studio/core/urls.py`
3. `label_studio/projects`
4. `label_studio/tasks`
5. `label_studio/io_storages`
6. `label_studio/ml`

### 前端

1. `web/README.md`
2. `web/apps/labelstudio`
3. `web/libs/editor`
4. `web/libs/datamanager`
5. `web/libs/ui`

### 部署

1. `Dockerfile`
2. `docker-compose.yml`
3. `deploy/docker-entrypoint.sh`
4. `deploy/uwsgi.ini`

## 5. 团队角色建议

如果是多人协作，建议至少按下面方式拆角色，不要所有人同时直接改全仓。

## 5.1 后端负责人

关注范围：

- Django app 结构
- API
- 数据模型
- 认证与权限
- 导入导出
- 存储接入

重点目录：

- `label_studio/projects`
- `label_studio/tasks`
- `label_studio/data_import`
- `label_studio/data_export`
- `label_studio/io_storages`

## 5.2 前端负责人

关注范围：

- 主产品页面
- 页面路由
- 设置页
- 管理界面
- 前后端联调

重点目录：

- `web/apps/labelstudio`
- `web/libs/app-common`
- `web/libs/ui`

## 5.3 标注器负责人

关注范围：

- 编辑器核心交互
- 标注组件
- 多模态标注体验
- 新标签能力

重点目录：

- `web/libs/editor`

这一块通常门槛最高，建议由最熟前端工程体系的人负责。

## 5.4 部署负责人

关注范围：

- Dockerfile
- Compose
- 环境变量
- Nginx
- uWSGI
- 数据目录与日志

重点目录：

- `deploy/`
- `docker-compose*.yml`
- `Dockerfile`

## 6. 二次开发时推荐的切入方式

## 6.1 如果目标是“做业务定制”

优先改这些：

- `web/apps/labelstudio`
- `label_studio/projects`
- `label_studio/tasks`
- `label_studio/data_import`
- `label_studio/data_export`

适合场景：

- 增加业务页面
- 增加项目配置项
- 增加字段或状态
- 修改导入导出流程
- 调整任务分发或列表行为

这类改造通常对现有架构冲击最小。

## 6.2 如果目标是“接入企业基础设施”

优先改这些：

- `label_studio/core/settings`
- `label_studio/io_storages`
- `deploy/`
- `docker-compose*.yml`

适合场景：

- 接入公司现有对象存储
- 接入公司认证体系
- 接入公司日志系统
- 接入内部网关、域名、证书、代理

## 6.3 如果目标是“改标注体验”

优先改这些：

- `web/libs/editor`
- `web/apps/playground`

适合场景：

- 新增标注标签
- 修改标注交互
- 改多媒体显示方式
- 增加快捷操作或视图能力

注意：

这一块虽然价值很高，但风险也最大，建议放在团队熟悉工程之后再做。

## 6.4 如果目标是“做 AI/LLM 协同标注”

优先改这些：

- `label_studio/ml`
- `label_studio/ml_models`
- `label_studio/ml_model_providers`
- 外部 `label-studio-ml-backend`

适合场景：

- 预标注
- 自动建议
- 模型对比
- LLM 评审
- 主动学习

## 7. 第一个迭代建议怎么做

建议团队的第一个迭代不要追求“全面平台化”，而是做一个最小闭环。

推荐的第一迭代范围：

1. 跑通本地开发和 Compose 部署
2. 新增一个你们自己的业务页面
3. 新增一个后端接口
4. 让页面调用接口成功
5. 补一条最小测试或验证路径

这样做的价值是：

- 能快速熟悉工程骨架
- 能验证团队协作流程
- 能尽早发现工程接入难点
- 风险可控，回滚也容易

不建议第一迭代就做：

- 编辑器核心重构
- 存储体系重构
- 权限体系全面改造
- 大规模前端换状态管理
- 大规模替换部署方式

---

## 8. 推荐的开发规范

## 8.1 后端改造建议

- 优先复用现有 app，不要轻易新建平行体系
- 接口改造先看已有 API 组织方式
- 配置改造优先走环境变量
- 涉及权限、用户、组织时要先确认现有模型关系
- 涉及导入导出时要关注异步任务和存储代理链路

## 8.2 前端改造建议

- 页面层改动优先放在 `web/apps/labelstudio`
- 通用组件尽量沉到 `web/libs/ui` 或 `web/libs/app-common`
- 编辑器改造不要和业务页面改造混在一个提交里
- React 页面新增时，不要忘记补 Django 路由入口

## 8.3 部署改造建议

- 不要直接改线上启动命令作为第一手段
- 优先调整环境变量和 compose 配置
- 先本地容器验证，再进入测试环境
- 数据目录、静态目录、数据库连接都要明确落盘位置

---

## 9. 推荐的验证方式

这个项目不适合只做“代码看起来没问题”的验证，建议至少保留下面三层验证：

## 9.1 命令级验证

- 后端是否能启动
- 前端是否能构建
- 容器是否能启动

## 9.2 页面级验证

- 登录是否正常
- 项目页面是否正常
- 目标改造页面是否正常
- 任务流是否受影响

## 9.3 数据级验证

- 数据导入是否正常
- 标注是否可保存
- 导出是否正常
- 如果涉及对象存储，代理读取是否正常

---

## 10. 上线前建议检查项

如果你们后续要把二次开发版本投入正式环境，建议上线前至少确认：

1. 数据库使用 PostgreSQL 或团队认可的 MySQL 方案
2. 不使用临时本地目录保存关键数据
3. 敏感配置全部走环境变量
4. 日志路径、异常日志、访问日志可追踪
5. 上传、导入、导出链路已经实测
6. 用户认证和权限链路已经验证
7. 对象存储、代理访问、跨域策略已确认
8. 至少做过一次备份恢复演练

---

## 11. 建议的接手节奏

如果你们是新团队接手，推荐按下面节奏推进：

### 第 1 阶段：看懂和跑通

- 跑通本地
- 跑通 Compose
- 读核心模块
- 明确改造目标

### 第 2 阶段：做最小闭环

- 只做一个业务点
- 页面 + 接口 + 验证
- 不做平台重构

### 第 3 阶段：进入正式二开

- 明确模块负责人
- 拆前后端和部署任务
- 逐步引入内部基础设施

### 第 4 阶段：再考虑平台深改

- 编辑器深改
- 存储深改
- 模型接入深改
- 高可用与规模化部署

---

## 12. 最后建议

这套工程适合作为“有现成基础、可快速落地、可持续扩展”的平台底座，但前提是团队接手方式要克制。

建议坚持三条原则：

1. 先跑通，再改造
2. 先做业务闭环，再做平台重构
3. 先按模块分工，再做跨层联动

如果这三条做到了，后续不管是做业务标注平台、模型评审平台，还是 AI 数据生产平台，都会顺很多。
