# Label Studio 前端页面二开本地开发与部署手册

## 1. 文档目标

这份文档专门面向“前端页面二开”场景，解决下面几个问题：

- 本地如何准备前端开发环境
- 后端和前端如何一起启动
- 改页面时具体要改哪些目录
- 新增页面时除了 React 代码还要补什么
- 本地改完后如何构建、如何用当前仓库方式部署验证
- Windows + PowerShell 下具体怎么执行

这份文档默认你当前工作目录是：

```powershell
E:\Label-Studio
```

---

## 2. 先看结论

如果你现在要二开前端页面，推荐使用下面这条开发路径：

1. 本地启动 Django 后端
2. 本地启动 `web` 前端开发服务
3. 浏览器访问 `http://localhost:8080`
4. 直接修改 `web/apps/labelstudio/src/pages` 相关代码
5. 改完后先本地联调验证
6. 再执行前端生产构建
7. 再通过 `docker build` 或 `docker compose up --build` 做本地部署验证

也就是说：

- 开发阶段不要直接靠 Docker 改代码
- 验证阶段再用 Docker

这是当前仓库成本最低、效率最高的方式。

## 3. 当前前端结构要先看懂

## 3.1 前端主目录

前端主工作区在：

- `web/`

核心目录如下：

- `web/apps/labelstudio`
  - 主产品前端应用
- `web/libs/editor`
  - 标注编辑器核心库
- `web/libs/datamanager`
  - Data Manager
- `web/libs/ui`
  - 公共 UI 组件
- `web/libs/app-common`
  - 应用级公共能力

如果你说的“前端页面二开”是普通业务页面、设置页、列表页、详情页，优先改：

- `web/apps/labelstudio`

## 3.2 页面入口

当前页面入口聚合文件是：

- `web/apps/labelstudio/src/pages/index.js`

这里维护了页面注册列表，例如当前仓库里能看到：

- `ProjectsPage`
- `OrganizationPage`
- `ModelsPage`
- `AccountSettingsPage`

因此：

- 修改已有页面，通常直接进对应页面目录
- 新增页面，通常需要把新页面注册到 `pages/index.js`

## 3.3 已有页面目录

当前 `web/apps/labelstudio/src/pages` 下主要有：

- `CreateProject`
- `DataManager`
- `ExportPage`
- `Home`
- `Organization`
- `Projects`
- `Settings`
- `WebhookPage`

如果你要先模仿一个现有页面，最适合参考的是：

- `web/apps/labelstudio/src/pages/Projects/Projects.jsx`

这个文件能直接看出：

- 页面是 React 组件
- 组件上会挂 `title`
- 组件上会挂 `path`
- 复杂页面还会挂 `routes`

## 4. 二开前需要准备什么

## 4.1 工具版本建议

建议本机至少满足：

- Python：`3.10` 到 `3.13`
- Node.js：建议 `22.x`
- Yarn：使用仓库锁文件安装
- Docker Desktop：最新版

说明：

- 仓库 `pyproject.toml` 要求 Python `>=3.10,<4`
- `Dockerfile` 默认使用 `Node 22`

## 4.2 安装后端依赖

在仓库根目录执行：

```powershell
pip install poetry
poetry install
```

## 4.3 安装前端依赖

在 `web` 目录执行：

```powershell
cd E:\Label-Studio\web
yarn install --frozen-lockfile
```

首次安装时间会比较长，属于正常情况。

## 5. 本地开发推荐启动方式

## 5.1 开两个终端

推荐开两个 PowerShell 窗口。

### 终端 1：启动后端

```powershell
cd E:\Label-Studio
$env:DJANGO_DB="sqlite"
$env:LOG_DIR="tmp"
$env:DEBUG="true"
$env:LOG_LEVEL="DEBUG"
$env:DJANGO_SETTINGS_MODULE="core.settings.label_studio"
poetry run python label_studio/manage.py migrate
poetry run python label_studio/manage.py runserver
```

启动后访问地址：

- `http://localhost:8080`

### 终端 2：启动前端开发服务

```powershell
cd E:\Label-Studio\web
yarn dev
```

前端 HMR 服务通常会跑在：

- `http://localhost:8010`

但你平时实际打开的仍然应该是：

- `http://localhost:8080`

因为：

- Django 是页面宿主
- 前端 dev server 负责热更新资源

## 5.2 可选：在根目录创建 `.env`

为了让 Django 正确走前端 HMR 资源，建议在根目录创建一个 `.env` 文件：

```env
FRONTEND_HMR=true
FRONTEND_HOSTNAME=http://localhost:8010
DJANGO_HOSTNAME=http://localhost:8080
```

注意：

- 当前仓库没有 `.env.development`
- 所以不要依赖 `Makefile` 里那个复制 `.env.development` 的步骤
- 直接手工创建 `.env` 即可

创建完 `.env` 后，建议重启 Django。

## 6. 你要改页面时，先判断是哪一类

## 6.1 只改已有页面 UI

适合场景：

- 改列表样式
- 改表单布局
- 改按钮位置
- 改标题、提示文案

这种情况下，通常只需要改：

- `web/apps/labelstudio/src/pages/...`
- 以及对应组件目录

一般不需要改 Django 路由。

## 6.2 新增一个 React 页面

适合场景：

- 新增业务页
- 新增设置页
- 新增独立面板

这种情况下，除了 React 页面本身，通常还要做两件事：

1. 把页面注册进前端页面体系
2. 在 Django 某个 `urls.py` 里补路由

这是当前仓库里最容易漏掉的一步。

## 6.3 改项目内嵌子页面

例如：

- `/projects/:id/data`
- `/projects/:id/settings/...`

这类页面通常不是单独在根页面列表里注册，而是挂在已有页面的 `routes` 或 `pages` 上。

参考：

- `web/apps/labelstudio/src/pages/Projects/Projects.jsx`

## 7. 修改已有页面的标准步骤

假设你要改项目列表页。

## 7.1 找到页面文件

例如：

- `web/apps/labelstudio/src/pages/Projects/Projects.jsx`

## 7.2 启动本地环境

按上面的两个终端方式启动。

## 7.3 修改代码

例如修改：

- 页面布局
- 标题
- 按钮
- 数据展示逻辑

保存后：

- 如果 HMR 正常，你会很快看到页面刷新
- 如果没有热更新，手动刷新 `http://localhost:8080`

## 7.4 验证接口联动

如果这个页面依赖接口，比如项目列表页会请求：

- `api/projects`

那就需要同时确认：

- Django 已正常启动
- 后端接口返回正常
- 浏览器控制台没有接口报错

## 8. 新增一个前端页面的标准步骤

下面给你一套更实际的操作方式。

## 8.1 先选一个页面集合

当前页面都在：

- `web/apps/labelstudio/src/pages`

如果是全新业务域，可以新建一个目录，比如：

- `web/apps/labelstudio/src/pages/CustomPage`

## 8.2 创建页面组件

例如新建：

- `web/apps/labelstudio/src/pages/CustomPage/CustomPage.jsx`

最小页面结构可以是：

```jsx
export const CustomPage = () => {
  return <div>Custom Page</div>;
};

CustomPage.title = "Custom Page";
CustomPage.path = "/custom-page";
CustomPage.exact = true;
```

## 8.3 创建页面目录出口

例如新建：

- `web/apps/labelstudio/src/pages/CustomPage/index.js`

内容：

```jsx
export { CustomPage } from "./CustomPage";
```

## 8.4 注册到页面入口

修改：

- `web/apps/labelstudio/src/pages/index.js`

把页面加进 `Pages` 列表。

例如：

```jsx
import { CustomPage } from "./CustomPage";

export const Pages = [
  isFF(FF_HOMEPAGE) && HomePage,
  ProjectsPage,
  OrganizationPage,
  ModelsPage,
  CustomPage,
  pages.AccountSettingsPage,
].filter(Boolean);
```

## 8.5 补 Django 路由

这是关键步骤。

因为当前 Label Studio 的页面访问最终还是经过 Django，所以如果你只写 React 页面而不补 Django 路由，直接访问新路径时容易 404。

补路由时，通常需要在某个后端 app 的 `urls.py` 里增加路径。

例如当前项目页路由在：

- `label_studio/projects/urls.py`

如果你的页面是某个业务模块下的页面，就应该补到对应模块的 `urls.py` 里。

如果只是内部已有页面路径扩展，也可以按现有路由模式扩展。

## 8.6 重启 Django 验证

新增路由后，建议：

1. 重启 Django
2. 保持前端 dev server 运行
3. 浏览器访问新路径

## 9. 页面开发时怎么选修改点

## 9.1 普通业务页面

优先改：

- `web/apps/labelstudio/src/pages`
- `web/apps/labelstudio/src/components`

## 9.2 通用组件

如果多个页面会复用，建议下沉到：

- `web/libs/ui`
- `web/libs/app-common`

## 9.3 不要一开始就改编辑器内核

如果你当前只是做页面二开，不建议一上来改：

- `web/libs/editor`

因为那不是普通页面层，而是标注器核心层，复杂度高很多。

## 10. 开发完成后，如何本地构建验证

开发阶段用 `yarn dev` 即可，但你最终还是要确认“生产构建产物是否正常”。

## 10.1 在 `web` 目录做前端构建

```powershell
cd E:\Label-Studio\web
yarn build
```

作用：

- 生成生产环境前端产物
- 验证你的页面修改是否能通过正式构建

如果这里失败，不要进入 Docker 阶段，先把构建问题解决掉。

## 10.2 运行前端单元测试

如果你的页面改动较多，建议至少执行：

```powershell
cd E:\Label-Studio\web
yarn test:unit
```

## 10.3 可选：运行主应用单元测试

```powershell
cd E:\Label-Studio\web
yarn ls:unit
```

## 11. 本地改完后如何“部署验证”

这里的“部署验证”分两种。

## 11.1 方式一：本地源码运行验证

这是开发阶段最推荐的方式。

流程：

1. Django 本地运行
2. 前端本地 dev server 运行
3. 浏览器访问 `http://localhost:8080`
4. 验证页面功能

优点：

- 最快
- 改一处看一处
- 最适合开发

## 11.2 方式二：本地容器部署验证

这是“我改完了，想模拟正式部署”的方式。

### 第一步：先构建前端和镜像

在项目根目录执行：

```powershell
cd E:\Label-Studio
docker build -t label-studio-local:frontend-dev .
```

说明：

- 这会按当前仓库 `Dockerfile` 执行完整多阶段构建
- 前端构建产物会被打进镜像

### 第二步：直接运行镜像

```powershell
docker run -it -p 8080:8080 -v ${PWD}\mydata:/label-studio/data label-studio-local:frontend-dev
```

然后访问：

- `http://localhost:8080`

### 第三步：或者使用 Compose 重建

如果你更想贴近当前仓库默认部署方式：

```powershell
cd E:\Label-Studio
docker compose up --build
```

这会：

- 重建镜像
- 启动 `app`
- 启动 `nginx`
- 启动 `db`

最终访问：

- `http://localhost:8080`

这一步更接近“本地部署验收”。

## 13. 典型问题与处理

## 13.1 页面文件改了，但浏览器没变化

优先检查：

- `yarn dev` 是否还在运行
- 根目录 `.env` 是否设置了 `FRONTEND_HMR=true`
- Django 是否重启过
- 浏览器是不是打开了 `http://localhost:8080`

## 13.2 新增页面访问 404

优先检查：

1. React 页面是否注册到 `web/apps/labelstudio/src/pages/index.js`
2. 页面组件是否设置了 `path`
3. Django 对应 `urls.py` 是否补了路由

很多时候就是第 3 条漏了。

## 13.3 `yarn build` 失败但 `yarn dev` 正常

这类问题很常见。

原因通常包括：

- 引用路径问题
- 生产构建下环境变量问题
- 某些动态导入或资源路径问题

处理原则：

- 以 `yarn build` 为准
- 因为最终部署依赖生产构建产物

## 13.4 Docker 里页面还是旧的

优先检查：

- 是否执行了 `docker build`
- 是否执行了 `docker compose up --build`
- 浏览器缓存是否清理
- 是否在看错误的容器或错误端口

## 13.5 挂载目录权限问题

如果容器启动时报写目录权限问题，要记住：

- 当前镜像默认不是 root 运行
- 数据目录权限必须可写

## 14. 你现在最适合怎么做

如果你现在就要开始二开前端页面，我建议按这个最短路径走：

1. `poetry install`
2. `yarn install --frozen-lockfile`
3. 根目录创建 `.env`
4. 启动 Django
5. 启动 `yarn dev`
6. 改 `web/apps/labelstudio/src/pages` 里的页面
7. 改完后执行 `yarn build`
8. 最后执行 `docker compose up --build` 做部署验收

这条路径最稳，也最符合当前仓库结构。
